# AniOS Cline Rules

These rules adapt [../AGENTS.md](../AGENTS.md) for a local coding model. They are an operational checklist, not a second engineering handbook. If these rules conflict with an explicit user instruction, follow the user instruction and report the conflict.

## 1. Load only relevant context

Before implementation or debugging, read in this order:

1. `AGENTS.md`
2. `docs/NEXT_SESSION.md`
3. `docs/ARCHITECTURE.md`
4. the relevant sections of `docs/DEVELOPMENT_GUIDE.md`

Also read:

- `docs/ROADMAP.md` for milestone or scope decisions;
- `docs/SECURITY.md` for secrets, personal data, permissions, authentication, network exposure, or external integrations;
- the relevant ADR for an architectural change.

Do not load every document automatically. Summarize the exact task, current evidence, acceptance criteria, first failing boundary, and first verification commands before editing.

## 2. Establish trustworthy evidence

For current behavior, use this priority:

1. runtime evidence from the current source revision or identified image;
2. current source and configuration;
3. automated test and build output;
4. `docs/NEXT_SESSION.md`;
5. architecture and roadmap documents;
6. earlier conversation context.

Before trusting runtime output, determine what is running. Check whether the process uses current host source, a bind mount, or an older built container. Record uncertainty as `UNVERIFIED`; never let an unidentified stale artifact override current source.

Use exactly these evidence labels:

- `VERIFIED`: the applicable acceptance path passed with concrete evidence.
- `FAILED`: the path was attempted and did not meet its criterion.
- `UNVERIFIED`: adequate evidence was not collected.

Mocks, interfaces, code generation, compilation, process startup, open ports, container status, health responses, and HTTP 2xx results are not functional verification by themselves.

## 3. Work one atomic loop

For each task:

1. Restate the objective and boundaries.
2. List what is `VERIFIED`, `FAILED`, and `UNVERIFIED`.
3. Define observable acceptance criteria.
4. Reproduce the first relevant failure.
5. Identify the first failing boundary.
6. Form one evidence-backed hypothesis.
7. Make the smallest targeted change.
8. Repeat the original acceptance path.
9. Run narrowly relevant regression checks.
10. Stop; do not automatically continue into the next failure or roadmap item.

A task may advance behavior labeled `SCAFFOLDED`, `MOCKED`, or `PLANNED` only when the user request or approved next atomic task explicitly targets that behavior.

## 4. Prevent debugging loops

Do not apply several speculative fixes together. Change one variable at a time.

After three unsuccessful targeted hypotheses on the same failure:

1. stop editing;
2. report the exact symptom and evidence;
3. list each hypothesis, change, and result;
4. revert failed experimental changes unless independently justified;
5. identify the missing evidence or smaller reproduction needed;
6. ask for guidance when further work requires a new decision or authority.

Do not escape a loop by redefining the task, lowering acceptance criteria, or calling partial behavior complete.

## 5. Validate the user-visible goal

Choose tests that prove the requested behavior, not implementation details.

For API work, validate as applicable:

- request schema and routing;
- dependency assembly and service entry;
- response status, headers, body, and stream termination;
- expected side effects and persisted records;
- intentional invalid-input behavior;
- absence of relevant server exceptions.

For UI work, endpoint reachability is only a prerequisite. `VERIFIED` UI evidence requires an automated browser test or documented manual browser session that:

1. opens the application in a real browser;
2. confirms the intended screen renders;
3. performs the user interaction;
4. observes required network requests;
5. checks the rendered result and state transitions;
6. fails on page exceptions or blocking console errors;
7. verifies loading, empty, success, and relevant failure states;
8. verifies reload or navigation behavior when persistence is part of the goal.

When browser tooling is unavailable, UI rendering and interaction remain `UNVERIFIED`. Do not substitute `curl`, HTML reachability, transformed JavaScript, or API-only tests.

For model-backed behavior, separately verify:

- deterministic application plumbing with a controlled fake provider in automated tests;
- real configured-provider integration in an explicit live acceptance check.

A fake provider can verify UI and orchestration behavior but cannot prove that a real LLM was called.

## 6. Follow dependency order

Investigate the earliest failing boundary first:

```text
infrastructure -> backend startup -> request validation -> dependency assembly
-> service -> model/streaming/persistence -> frontend API client -> browser UI
```

Do not debug downstream behavior while an upstream prerequisite is failing, unless gathering downstream evidence is explicitly required.

## 7. Control scope and commands

- Use service names, ports, routes, schemas, environment variables, test paths, and commands from the repository and `docs/DEVELOPMENT_GUIDE.md`.
- Add one brief, plain-language purpose comment immediately above every function or method you write, including helpers, handlers, frontend functions, tests, CLI commands, and migrations. Place it above decorators and keep it accurate when behavior changes.
- On Windows, do not use Bash line continuations in PowerShell. Prefer `Invoke-WebRequest`, `Invoke-RestMethod`, or explicit `curl.exe`.
- Do not add dependencies, alter public interfaces, refactor unrelated code, delete data, reset databases, remove volumes, discard work, or regenerate configuration without appropriate scope or approval.
- Preserve pre-existing changes.
- If Git is available, record `git branch --show-current`, `git rev-parse HEAD`, and `git status --short` before editing and again at handoff.
- Do not stage or commit unrelated user changes. Do not create commits, tags, branches, worktrees, reverts, or stashes unless the user or requested workflow authorizes them.
- A commit is a verified checkpoint only when that exact tree passed the functional acceptance path. Record its SHA and evidence in `docs/NEXT_SESSION.md` after verification.
- Prefer a separate recovery branch or worktree at a verified SHA over overwriting the current worktree.
- Never automatically run `git reset --hard`, `git clean -fd`, `git restore .`, destructive checkout commands, or force pushes.
- If Git is unavailable, report `UNAVAILABLE`; do not invent branch, status, diff, or commit information.

## 8. Maintain documentation without churn

For implementation, debugging, or validation work:

- Rewrite `docs/NEXT_SESSION.md` when evidence, blockers, last action, or the next atomic task changed.
- Append to `docs/CHANGELOG.md` only after meaningful behavior is functionally `VERIFIED`.
- Update `docs/ROADMAP.md` only when milestone evidence or scope changed.
- Update `docs/ARCHITECTURE.md` only when components, boundaries, ownership, or data flow changed.
- Update `docs/DEVELOPMENT_GUIDE.md` only when setup, commands, dependencies, tests, or recurring procedures changed.
- Update `docs/SECURITY.md` only when security facts or required controls changed.
- Create an ADR only for a significant architectural decision.
- For every modifying task, assess diagram impact against both the full-system view and every detailed subsystem view that owns the changed code.
- Update each affected source and rendered artifact in `docs/diagrams/` only when components, agents, persistent stores, external dependencies, deployment/trust boundaries, ownership, or cross-component data flow changed.
- When an architectural subsystem has no detailed view, add its `.mmd`/`.svg` pair and register it in the diagram catalog and renderer suite.
- Report exactly `Diagram impact: UPDATED — <diagram names>` or `Diagram impact: NONE — <reason>` for every modifying task.

Do not modify documentation during a read-only review or when the user forbids changes. Report stale documentation instead.

Before ending a modifying task, check internal links, deleted-file references, stale status claims, commands, ports, routes, and conflicting maturity labels.

## 9. Completion report

Report:

- objective and scope;
- files created, modified, and deleted;
- commands executed;
- every acceptance criterion as `VERIFIED`, `FAILED`, or `UNVERIFIED` with evidence;
- functional and regression results;
- documents reviewed and changed;
- diagram impact with the exact updated diagram names or a concrete reason no diagram changed;
- remaining issues and the single next atomic task;
- Git state when available, or `UNAVAILABLE` when it is not.

Do not report `COMPLETE` while an acceptance criterion for the atomic task is failed or unverified. A task that only changed code without proving its intended behavior is not complete.
